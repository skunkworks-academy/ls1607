# Mentoring, Feedback and Community

Luca explicitly identified lack of direction and lack of cybersecurity peers as major obstacles. Mentoring is therefore a core control in the roadmap, not an optional benefit.

## Mentoring model

| Element | Requirement |
|---|---|
| Primary mentor | Cybersecurity practitioner able to review foundations and practical work |
| Cadence | Fortnightly for the first 90 days, then at least monthly |
| Session duration | 30–45 minutes |
| Review method | Evidence first: repositories, lab records, diagrams and reports |
| Feedback preference | Confirm during validation; use written actions after every session |
| Escalation | Career consultant or training lead when progression, welfare or scope requires attention |

## First six mentor sessions

| Session | Focus | Output |
|---:|---|---|
| 1 | Validate IDR, time allocation and priorities | Approved first 30-day plan |
| 2 | Review Linux, network and GitHub baselines | Targeted remediation list |
| 3 | Review OpenClaw architecture and data flow | Initial risk register |
| 4 | Review threat model and minimum controls | Security actions and owners |
| 5 | Review first public repository | Documentation and code feedback |
| 6 | Conduct 90-day review | Stage 2 decision and credential direction |

## Peer and community exposure

Target at least one meaningful peer interaction each month:

- Skunkworks Academy cybersecurity cohort or project team
- TryHackMe learning community activity
- GitHub discussion, issue or code review
- Local or virtual cybersecurity event
- ADF communications or veteran career discussion where appropriate
- Student cybersecurity group or degree-preparation community

:::caution Community safety
Luca is 19 and early in his career. Public community participation must avoid sharing private contact details, credentials, home-network data or operational information. Mentors should review public technical content until safe publishing habits are established.
:::

## Feedback quality standard

Feedback should be:

- **Specific:** linked to exact evidence or behaviour
- **Actionable:** defines what should change
- **Prioritised:** separates critical gaps from optional improvement
- **Balanced:** recognises progress while maintaining standards
- **Recorded:** ends with owners and dates
- **Developmental:** supports independence rather than permanent dependence
